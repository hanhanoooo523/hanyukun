StyleShift

Design presentation — browser-based style transfer & regional editing

This deck focuses on why we designed the tool, how it supports visual communication workflows, our design thinking, and key UI/UX decisions. Development is noted only briefly.

Motivation

Why we made this tool

  • Gap in the loop. Designers often mood-board and sketch variations in separate tools; rapid, on-image style experimentation is fragmented across apps and plugins.
  • Communication, not novelty. Brief exploration helps align clients and teams on tone (warm vs. cold, analog vs. digital) before high-fidelity production.
  • Control where it matters. Global presets are not enough — local adjustments matter for brand marks, skin tones, or hero objects without redoing the whole comp.

We built StyleShift as a single-surface workshop: upload → explore styles → compare → refine by region.

Workflow fit

Use in visual communication design

  • Early ideation: Quick passes (oil, ink, cyberpunk, etc.) to stress-test art direction on real layout imagery.
  • Client / stakeholder dialogue: Split and compare views make “before / after” legible in review — less abstract than filter names alone.
  • Iterative refinement: After a global pass, element decomposition supports “only this area” restyling — closer to how designers think in layers and selections.
  • Documentation: Export and history strip support capturing options for decks and brand guidelines.

Design thinking

How we approached the problem

  • Empathize: Personas skewed toward students and designers who work in browsers, need low friction, and lose context when jumping between tools.
  • Define: Core job — “see many believable style directions on my image without leaving one UI.” Secondary — “fix one region without breaking the rest.”
  • Ideate: Explored layouts (single canvas vs. triptych); chose a three-column mental model matching source → controls → output.
  • Prototype & test: Paper → low-fi wireframes → interactive HTML; tightened affordances (sliders, compare handle, purple selection overlay) through use.

UI / UX — Information architecture

Three-column layout

  • Left — Source. Upload and history stay anchored; users never lose “where the original lives.”
  • Center — Control. Styles, parameters, tint, view mode, and primary action (generate) form one vertical pipeline — reduces hunting.
  • Right — Result. Largest share of attention after generation; secondary tools (regions) sit below the canvas so the image stays primary.

Why: Matches conventional design software (assets / inspector / canvas) and supports teaching walkthroughs left-to-right.

UI / UX — Feedback & trust

Compare, split, and preview modes

  • Compare slider: Draggable seam — same framing for both states; avoids eye movement between two windows.
  • Split view: Full-width original vs. result — when separation clarity matters more than a single composite.
  • Aspect-safe rendering: Comparison uses contained scaling so the image is not crushed — truthful preview reinforces trust.

UI / UX — Progressive disclosure

Element decomposition & regional controls

  • Progressive complexity: Region tools appear only after a result exists — avoids overwhelming first-time users.
  • Color-region metaphor: Chips + optional “connected patch” click mirrors how designers think about dominant colors and masks.
  • Regional parameters: Separate sliders + “sync from center” honor the case where local treatment should diverge from global style strength.
  • Pick mode vs. pan: Explicit toggle reduces accidental moves when precision selection matters.

UI / UX — Ergonomics & access

Resizable stack, undo, language

  • Resizable divider between preview and tool panel: small screens and long forms differ; users own their vertical real estate. Preference persists locally.
  • Undo at the header: Global, discoverable safety net for experimental edits (generate and regional ops).
  • Bilingual UI (ZH / EN): Supports classroom and portfolio contexts without forking the product.
  • Visual language: Dark surface, violet accent, restrained chrome — keeps focus on the image, not the chrome.

UI / UX — Micro-decisions

Details that support the story

  • Toast + progress — lightweight confirmation without modal interrupt.
  • Outlined selection (purple) — readable on dark UI and varied imagery.
  • Zoom on result — inspection before export; mimics designer zoom habits.
  • Style cards — icon + short label + micro-copy to disambiguate presets faster than a dropdown alone.

Development (brief)

What we built on

The prototype is implemented as a single-page HTML/JS canvas pipeline for portability (open file, demo in class, host statically). Engineering choices served design goals: no install friction, fast iteration on layout, and full control over the compare/split and region compositing behaviors.

Implementation supports the UX; it is not the subject of this deck — the design rationale above is.

StyleShift is a design-first tool: it shortens the loop between reference and styled outcome, makes comparison explicit, and adds regional control where global presets fail communication goals.
